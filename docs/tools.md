# Semaphore Tools
Semaphore has a tools folder containing tools for monitoring how the semaphore system is performing across our development and production servers.

## parse_semaphore_logs.sh
`parse_semaphore_logs.sh` is a bash utility that scans Semaphore model log files 
for a given date range and produces a CSV summarizing each model run — whether it 
succeeded, failed, and why.

### Usage
```bash
./tools/parse_semaphore_logs.sh  
```

Dates must be in `MM/DD/YY` format. For example:
```bash
./tools/parse_semaphore_logs.sh 02/07/26 02/18/26
```

The script can be run from anywhere in the repository. It resolves all paths relative 
to its own location, so you don't need to `cd` anywhere first.

### How It Works
1. **Selects log files by date range** — rather than scanning all logs, it builds 
   `YYYY_M_` filename patterns from your date range (e.g. `2026_2_*.log`) and only 
   opens files that could contain relevant entries. This keeps it fast even with 
   hundreds of model log directories.

2. **Parses each file line by line** — it looks for three kinds of log events:
   - Lines containing `completed successfully` → recorded as a `success`
   - Lines containing `DateRangeValidation:` → signals the start of a failure block, 
     and the script begins accumulating context lines
   - Lines containing `FAILED - Null result inserted` → closes the failure block and 
     writes a `failed` row, extracting the reason and details from the accumulated context

3. **Writes results to a timestamped CSV** saved in `parsed_semaphore_logs/` at the 
   repo root. The folder is created automatically if it doesn't exist.

4. **Prints a summary** to stdout when finished — total entries, success/failure 
   counts, success rate, top failure reasons, and which models failed most often.

## Database Migrations

Database versions are managed using `tools/migrate_db.py`.

To initialize or update the database:

```bash
docker exec semaphore-core python3 tools/migrate_db.py
```

See the **Database** documentation for information on creating new migrations.

## group_runner.py

`group_runner.py` is a helper script used by the cron scheduler. It reads one of the intermediate JSON files created by `init_cron.py` and runs all of the DSPECs listed in that file with a single Semaphore command.

Normally, you will not need to run this script directly since it is called automatically by the cron jobs generated by `init_cron.py`.

### Usage

Run the script by providing the path to an intermediate JSON file.

```bash
python3 tools/group_runner.py -i ./cron_jobs/job_300_3600_0.json
```

### What the Tool Does

1. Reads the specified intermediate JSON file.
2. Loads the list of DSPEC paths stored in the file.
3. Builds a Semaphore command containing all of the DSPECs.
4. Executes the command inside the `semaphore-core` Docker container.

The generated command looks similar to:

```bash
docker exec semaphore-core python3 src/semaphoreRunner.py -d \
./data/dspec/model1.json \
./data/dspec/model2.json \
./data/dspec/model3.json
```

### Notes

- This tool is intended to be used by the cron scheduler and typically does not need to be run manually.
- The intermediate JSON files must already exist. They are generated by `init_cron.py`.
- Make sure the `semaphore-core` Docker container is running before executing this tool.

## init_cron.py

`init_cron.py` generates the cron schedule used to run Semaphore models automatically. Instead of manually creating cron jobs, this tool scans all DSPEC files, groups models with the same schedule together, and generates the cron entries for you.

Models are grouped using the `interval` and `offset` values from each DSPEC's `timingInfo`. Within each group, models are sorted by lead time (largest first) so the models that request the most data run before the shorter lead-time models.

### Usage

Run the tool by providing the DSPEC directory and a folder where the intermediate job files should be written.

```bash
python3 tools/init_cron.py \
    --root_directory ./data/dspec \
    --intermediate_file_folder ./cron_jobs
```

### Optional Arguments

| Argument | Description | Default |
|----------|-------------|---------|
| `--max_dspec_per_command` | Maximum number of DSPECs included in a single cron job. | `999` |

### What the Tool Does

1. Recursively scans the DSPEC directory.
2. Reads each model's timing information.
3. Skips inactive models.
4. Groups models that share the same interval and offset.
5. Sorts each group by lead time.
6. Creates intermediate JSON files containing the grouped DSPEC paths.
7. Generates a `semaphore.cron` file.
8. Replaces the current system crontab with the newly generated one.

### Generated Files

#### `semaphore.cron`

Contains the cron schedule used to run Semaphore.

Example:

```cron
5 */1 * * * cd semaphore && python3 ./tools/group_runner.py -i ./cron_jobs/job_300_3600_0.json
```

#### Intermediate Job Files

The intermediate directory contains JSON files listing the DSPECs that should be run together.

Example:

```text
cron_jobs/
├── job_300_3600_0.json
├── job_300_3600_1.json
└── job_1200_21600_0.json
```

Each file contains a JSON array of DSPEC paths that is passed to `group_runner.py`.

### Notes

- Only active DSPECs (`timingInfo.active = true`) are included in the generated cron schedule.
- Running this tool replaces the current system crontab with the newly generated one.

```bash
crontab -r
crontab ./semaphore.cron
```

- If you add a new DSPEC or change a model's `interval`, `offset`, or `active` flag, rerun `init_cron.py` so the cron schedule stays up to date.
